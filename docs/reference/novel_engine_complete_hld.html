Main orchestrator (Python FSM)
Stateful finite state machine. States: INIT → OUTLINE → CHAPTER_PLAN → SCENE_WRITE → SCENE_EDIT → MEMORY_UPDATE → CHECKPOINT → NEXT_SCENE. Drives all agent calls. Manages checkpoint file. Handles fallback to Ollama if Hypura errors.

Inference layer — two servers, always running
Server A — Phi-3.5 Mini via Ollama (structural brain)
Port 11434. Handles all structured JSON tasks: story bible generation, three-act outline, chapter plans, scene beats, coherence checks, timeline validation, bible updates. Runs entirely in GPU Metal. Near-instant. Never touches NVMe.

Server B — Mixtral 8x7B or Llama 70B via Hypura serve (prose engine)
Port 11435 (Ollama-compatible API). Handles only prose generation and literary editing. Hypura schedules attention layers to GPU, overflow to RAM, remaining weights streamed from NVMe. MoE expert streaming for Mixtral gives 99.5% cache hit rate. Zero overhead if model fits in GPU+RAM.

Memory layer
Hierarchical RAG — 3-tier retrieval (ChromaDB)
Tier 1: Novel-level summary, always injected (~300 tokens). Tier 2: 3 adjacent chapter summaries, fetched by position (~600 tokens). Tier 3: Top-5 semantically similar scene chunks, fetched by cosine similarity (~800 tokens). Total injection budget capped at 1700 tokens regardless of novel length. Embeddings via nomic-embed-text (local, Ollama).

Agent layer — 5 roles, same models
Architect · Planner · Editor · Memory Manager → Phi-3.5 Mini
All structured roles use the fast structural model. Architect builds the story bible and three-act outline. Planner expands each chapter into 4–8 scene beats with plot thread tracking. Editor reviews completed prose for coherence, not style. Memory Manager compresses scenes into summaries and updates the bible after each chapter.

Writer → Mixtral 8x7B / Llama 70B
Prose generation only. Receives: scene beat from Planner, RAG context (1700 tokens), character voice guide, genre style prefix. Produces 600–1500 words per scene. Temperature 0.75. Top-p 0.92. No meta-commentary, no scene headings — raw prose only. Hypura port 11435.

Resilience layer
Checkpoint + crash recovery
JSON checkpoint file written after every completed scene: {chapter_idx, scene_idx, bible_version, rag_index_hash, completed_prose_path}. On startup, orchestrator checks for checkpoint and resumes from last scene boundary. Hypura prevents OOM crashes; checkpointing covers everything else (power, sleep, network).

Fallback inference
If Hypura serve fails to load the prose model (new GGUF, edge case), the orchestrator falls back to Ollama + Mistral 7B Q5_K_M automatically. One config flag. Output quality degrades but generation does not stop. Error logged with chapter/scene for later re-generation.
config.yaml flagMistral 7B fallbackauto-resume

Output pipeline
Each scene written as Markdown. Chapter stitcher merges scenes with scene breaks. Pandoc converts to EPUB, DOCX, or PDF. Metadata sidecar: character index, unresolved plot threads, timeline, word count per chapter. Genre-specific front matter (world map stub for fantasy, cast list for thriller).

Structural model — same for both hardware tiers
Phi-3.5 Mini Instruct 3.8B · always via Ollama recommended
phi3.5:3.8b-mini-instruct-q5_K_M
~2.3 GB · Metal GPU
Microsoft's best small model. Exceptional at structured JSON output — critical for story bible and agent communication. Strong instruction following. Context window: 128K tokens. Runs entirely in GPU RAM leaving headroom for the prose model. Runs at ~25 tok/s on M1 — fast enough that structural phases feel instant.
Alternative: Qwen2.5-3B-Instruct-Q5_K_M
~2.0 GB · Metal GPU
Smaller footprint, nearly identical JSON output quality. Good if you want to keep the structural model's RAM footprint minimal to leave more room for a larger prose model.
Prose model — 8 GB M1 Air
8 GB Air — Hypura MoE streaming is the right choice
Mixtral-8x7B-Instruct-v0.1-Q4_K_M
~26 GB · Hypura MoE stream
best for 8 GB MoE architecture = only 2 of 8 experts active per token. Hypura's 99.5% expert cache hit rate makes this surprisingly fast. Effectively ~47B parameter quality for prose, active at ~13B parameters per step. Confirmed working by Hypura documentation. Creative prose quality far exceeds any 7B or 8B model.
Mistral-7B-Instruct-v0.3-Q5_K_M
~5.1 GB · Ollama Metal GPU
fallback only Fits entirely in GPU RAM. Use as the fallback if Hypura has issues with a specific GGUF. Good prose for a 7B model but noticeably lower ceiling than Mixtral.
Prose model — 16 GB M1 Air
16 GB Air — three strong options, ranked by quality
Qwen2.5-72B-Instruct-Q4_K_M
~40 GB · Hypura NVMe stream
best prose quality Qwen2.5-72B is widely considered the best creative writing model at this size class — superior narrative voice, character consistency, and instruction following over Llama 70B. Dense model: Hypura uses FFN streaming via pread(). Slow (~0.3 tok/s) but since speed is not a concern, this produces the highest quality output available on consumer hardware.
Llama-3.1-70B-Instruct-Q4_K_M
~40 GB · Hypura NVMe stream
strong alternative Very strong structural reasoning and plot coherence. Slightly better than Qwen2.5-72B at following complex multi-constraint prompts (like "write in the voice of character X, referencing event Y, while advancing thread Z"). Different strength profile than Qwen — try both and choose based on your genre.
Mixtral-8x7B-Instruct-v0.1-Q5_K_M
~32 GB · Hypura MoE stream
faster option On 16 GB, Mixtral 8x7B runs at ~2.2 tok/s (confirmed). If you want a balance of quality and speed, this is it. Prose quality is excellent — just below the 70B tier. Use Q5_K_M for better quality than Q4_K_M with only ~6 GB more disk.
Embedding model — for RAG memory
nomic-embed-text v1.5 recommended
Pulls via Ollama. 137M parameters, 768-dim embeddings, 8192 token context. Purpose-built for document retrieval. Runs in CPU/GPU, ~150 MB RAM. Significantly better recall than all-MiniLM-L6 for longer narrative passages. Pull with: ollama pull nomic-embed-text
What TurboQuant (imatrix-calibrated GGUF) achieves
Standard quantization treats all weights equally — it reduces every weight's precision by the same amount. imatrix quantization first measures which weights matter most for YOUR specific use case (literary text), then protects those weights during quantization. Result: measurably lower perplexity loss vs naive Q4_K_M, especially for domain-specific creative tasks. This is the best quality-per-GB you can get offline.
Step-by-step quantization pipeline
1
Install llama.cpp with Metal support
Build from source to get the latest quantization tools. Homebrew formula may lag behind — source build ensures imatrix support is present.
brew install cmake && git clone https://github.com/ggerganov/llama.cpp && cd llama.cpp && cmake -B build -DLLAMA_METAL=ON && cmake --build build --config Release -j8
2
Download base models in full precision (BF16)
Download from HuggingFace in safetensors format. You need the unquantized base model to quantize it yourself. Use huggingface-cli. Mixtral-8x7B-Instruct-v0.1 is ~90 GB in BF16 — needs external SSD temporarily. If disk is a concern, download a pre-quantized Q8_0 GGUF (~47 GB) instead and re-quantize from that.
pip install huggingface_hub && huggingface-cli download mistralai/Mixtral-8x7B-Instruct-v0.1 --local-dir ./models/mixtral-bf16
3
Convert to GGUF format (if starting from safetensors)
llama.cpp's convert script produces a F16 GGUF ready for quantization. Skip if you downloaded a pre-made GGUF.
python llama.cpp/convert_hf_to_gguf.py ./models/mixtral-bf16 --outfile ./models/mixtral-f16.gguf --outtype f16
4
Build literary calibration dataset
The imatrix is only as good as the calibration text. Download 50–100 public domain novels from Project Gutenberg across your target genres. Concat into a single calibration.txt (~10–20 MB). Include: literary fiction, thriller, fantasy, romance, sci-fi. Diverse literary text produces a better imatrix for novel writing than the default code/wiki datasets.
python scripts/build_calibration.py --gutenberg-ids 1342,11,84,2701,1661 --output calibration.txt
5
Generate importance matrix (imatrix)
Run the model over the calibration dataset and record which weights activate most. Takes 30–90 minutes. The --chunks flag controls how much of the calibration data is processed — 256 chunks is thorough.
./build/bin/llama-imatrix -m ./models/mixtral-f16.gguf -f calibration.txt -o ./models/mixtral-imatrix.dat --chunks 256 -ngl 99
6
Quantize with imatrix (Q4_K_M for 8 GB · Q5_K_M for 16 GB)
The --imatrix flag tells llama-quantize to use the importance data. Q4_K_M gives ~26 GB for Mixtral. Q5_K_M gives ~32 GB. For Phi-3.5 Mini use Q5_K_M (~2.3 GB) — the small size means you can afford the quality upgrade.
./build/bin/llama-quantize --imatrix ./models/mixtral-imatrix.dat ./models/mixtral-f16.gguf ./models/mixtral-8x7b-imatrix-Q4_K_M.gguf Q4_K_M
7
Validate with perplexity benchmark
Run perplexity on a held-out literary text. Compare your imatrix Q4_K_M against a naive Q4_K_M — you should see 5–15% lower perplexity. If similar, the calibration dataset may need more genre diversity.
./build/bin/llama-perplexity -m ./models/mixtral-8x7b-imatrix-Q4_K_M.gguf -f validation_text.txt -ngl 99
8
Load into Hypura and profile
Run hypura profile once after first install, then test your quantized model with --max-tokens 10 before a full run. Use hypura inspect to see how it plans to distribute tensors across GPU/RAM/NVMe.
hypura profile && hypura inspect ./models/mixtral-8x7b-imatrix-Q4_K_M.gguf && hypura serve ./models/mixtral-8x7b-imatrix-Q4_K_M.gguf --port 11435
